# CLAUDE.md — eco-biome-visualizer

Developer memory for this repo. Read this before making changes.

## What this is

A single self-contained page that reads an Eco `WorldGenerator.eco` config and charts where each
material spawns by biome and world height (Y). No build step, no dependencies, no network calls
except fetching the bundled default config. Published via GitHub Pages.

## Files

- `index.html` — **everything** (HTML + CSS + vanilla JS in one file). This is the whole app.
- `WorldGenerator.eco.template` — Eco's default world config. Fetched on page load as the default chart.
- `README.md` — user-facing docs.

There is no framework, bundler, or package.json. Edit `index.html` directly.

## Run / preview

- **Served over http/https** (GitHub Pages, or `python -m http.server`): the default world auto-loads.
- **Opened as `file://`**: browsers block `fetch`, so the default world does NOT auto-load — you get the
  drop/paste UI instead. This is expected. Always test the default-load path over HTTP.

## Pipeline (all in `index.html`)

`load(text)` → `extract(cfg)` → per entry `depthProfile()` → `smearY()` → `render()`; hover uses the
precomputed profiles.

1. **`extract(cfg)`** — parses the config into an array of `{bi, ore, surf:[loY,hiY], on, mods:[...]}`:
   - Builds an `$id` map and `deref()`s the `$id`/`$ref` reference graph (Newtonsoft PreserveReferences format).
   - Finds the Voronoi weights object by searching for `CoolForestWeight`/`DesertWeight` (used for the `on`
     "does this biome generate" flag).
   - Reads `WaterLevel` / `MaxGenerationHeight` from the config; surface Y = `WL + elev*(MG-WL)` using the
     per-biome normalized elevation bands in `ELEV` (hard-coded from the engine's `Biomes.cs`).
   - Walks `TerrainModule.Modules` (one `BiomeTerrainModule` per biome) → `Module.BlockDepthRanges` →
     `SubModules`. Each tracked material becomes a mod:
     - `StandardTerrainModule` → `{t:'std', a,b, w}` where `w`=PercentChance over depth `[a,b]`.
     - `DepositTerrainModule` → `{t:'dep', sa,sb (seed DepthRange), ba,bb (body DepositDepthRange), w=spawn*avgBlocks, bo (aspect boost), N (avg blocks)}`.
     - Primary stratum block (`BlockDepthRange.BlockType`, mainly clay/peat) → `{t:'strat', a=Min,b=Max, w:0.6}` (approximation).
2. **`depthProfile(e)`** — sums each mod into a per-depth array (`DMAX` deep). `std`/`strat` are flat over
   their band. `dep` is spread with an area-normalized triangular kernel whose half-width is the estimated
   vertical extent `ey = 0.62 * cbrt(N) * bo`, clipped to the body range.
3. **`smearY(dp, surf)`** — convolves the depth profile across the biome's surface-Y range to get an
   absolute-Y profile (a fixed-depth vein "smears" across the biome's elevation spread).
4. **`render()`** — draws grouped violins/bars; `gMax` = global max across all profiles for the hover index.

## Config tables = source of truth (edit these to change coverage)

All near the top of the `<script>`:

- `MATS` — `[blockTypeSubstring, materialKey]` pairs. `material()` returns the first substring that matches
  the block's `Type`. Crushed variants map to the same key. **Order:** more specific first is safest.
- `COL` — material → hex color.
- `ON` — material → display label.
- `oreOrder` — material draw/legend/sort order. Legend is generated by iterating this.
- `ELEV` — biome → `[minElev, maxElev]` normalized (from `Biomes.cs`). Drives surface Y.
- `WEIGHTF` — biome → Voronoi weight field name (for the `on` flag).
- `ALWAYS` — biomes always treated as present (Grassland, coasts).
- `DISP` — biome → display name.
- `biomeOrder` — biome column/sort order.

### To add a material

Add it to `MATS`, `COL`, `ON`, and `oreOrder`. That's all — grouping, legend, sorting, hover pick it up
automatically. (Sulfur and Peat were added exactly this way.)

### To add / rename a biome

Add to `ELEV`, `DISP`, `biomeOrder`, and (if weight-driven) `WEIGHTF`. If it's a coast/always-on biome,
add to `ALWAYS`. Biomes not in `ELEV` are skipped.

## Density model (keep it honest)

- The value is a **relative index**, not blocks-per-chunk. Deposits use `w = spawn * avgBlocks`, conserved
  and spread by `ey`. A lateral deposit (low Y weight → small `ey`) reads short & dense; a vertical one
  reads tall & thin. Isotropic weights give `bo ≈ 1` (no change).
- `aspect boost bo = wy^(2/3) / (wx*wz)^(1/3)` from the **mean** of `DirectionWeights` (real spawner picks
  one vector at random per deposit — so this is an approximation).
- Violin widths are normalized **per column** and drawn on a √ (perceptual) scale. The **hover index is
  linear and global** (`value/gMax*100`) — that's the only fair cross-column/material comparison.
- Clay/peat strata are flat-band approximations; true stratum selection is noise-driven and cumulative.

## Gotchas (learned the hard way)

- **Never use a bare global `var top` / `name` / etc.** — they alias read-only `window.*` properties in a
  non-module script and throw on assignment. The layout constant is `padTop` for this reason.
- **LF line endings.** Keep them; don't let an editor rewrite to CRLF.
- **`file://` blocks the default-world fetch** (see Run/preview). Graceful fallback is intentional.
- The chart can get **wide in group-by-biome mode**; groups have a min slot width (`MINGW`) so headers like
  "Warm coast" don't overlap. Adding materials widens it further (it scrolls horizontally if needed).

## Data source

The terrain module (block strata / ore depths) is Eco's standard world generator. Reference configs live
in the Eco repo at `Server/Configs/WorldGenerator.eco` and `WorldGenerator.eco.template`. Only seed and
Voronoi weights differ between worlds; the ore-vs-depth strata are the same standard module.

## Verifying a change without a browser

The `extract`/profile logic is plain JS. To sanity-check parsing against a real config, port the relevant
functions into a Node script and run them against `WorldGenerator.eco.template` (or any `.eco`), printing
the resulting entries. Also `node --check` the extracted `<script>` body for syntax.
